Build a production-minded MVP called **Sign Takeoff Portal** for the sign industry.

## Product Goal
This is a portal for sign contractors, sign shops, and fabricators. A user uploads plan PDFs, and the system uses Gemini to read the plans, identify sign-related information, extract structured takeoff data, and export the results into Excel. This is not a generic construction takeoff app. It is specifically for signage.

The system should understand things like:
- sign schedules
- sign callouts
- sign types
- quantities
- sign locations
- dimensions
- mounting types
- finishes
- illumination notes
- materials when available
- message/content when available
- general notes related to signs
- uncertain or missing items that need review

In a later phase, the system will also retrieve sign code requirements by jurisdiction using a RAG knowledge base, so the architecture should be designed for that now.

## Tech Stack
Use this stack:
- Frontend: React + Vite + TypeScript
- Backend: Python + FastAPI
- AI model: Gemini
- Embeddings: current supported Gemini embedding model only, no deprecated embedding models
- Vector database: Chroma with persistent local storage
- File export: XLSX
- Storage:
  - uploaded files stored locally in app storage
  - parsed outputs stored locally
  - Chroma db stored locally
- Keep the code modular and production-friendly

## Core MVP Features
Plan the app to include these features:

### 1. Upload Flow
- User uploads one or more PDF plan files
- Backend stores uploaded files
- Backend creates a processing job

### 2. Plan Extraction
Use Gemini to analyze uploaded PDFs and extract sign-specific data into structured JSON.

Each extracted sign row should try to include:
- sheet number
- detail/reference if available
- sign type
- sign identifier/callout
- quantity
- location
- dimensions
- mounting type
- finish/color
- illumination
- materials
- message/content
- notes
- confidence score
- review flag

### 3. Review Screen
- Frontend should show extraction results in a table
- User should be able to see which rows are high confidence and which need review
- Rows with missing or uncertain data should be visibly flagged

### 4. Excel Export
- Export reviewed structured results to XLSX
- Keep the export schema clean and consistent

### 5. RAG-Ready Knowledge Layer
Even if not fully implemented yet, architect for this:
- Chroma persistent database
- collections for:
  - federal_codes
  - state_codes
  - city_codes
  - sign_glossary
  - plan_guides
  - customer_standards
- metadata on chunks:
  - source_file
  - jurisdiction
  - doc_type
  - section
  - effective_date
  - status
- retrieval layer should be designed to support jurisdiction-filtered lookups later
